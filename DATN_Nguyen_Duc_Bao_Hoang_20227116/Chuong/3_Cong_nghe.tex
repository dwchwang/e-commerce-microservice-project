\section{Tổng quan giải pháp công nghệ}
\label{section:3.1}

Hệ thống được xây dựng dựa trên một tập hợp công nghệ được tuyển chọn để đáp ứng đồng thời
các yêu cầu về kiến trúc microservices, xử lý phân tán, bảo mật, hiệu năng và khả năng vận
hành. Ở tầng ngôn ngữ và nền tảng, Java 21 kết hợp với Spring Boot 3.5 và Maven multi-module
tạo thành bộ khung phát triển chính. Ở tầng điều phối, Spring Cloud cung cấp ba thành phần
cốt lõi: API Gateway (Spring Cloud Gateway), Service Registry (Netflix Eureka) và Config
Server (Spring Cloud Config), cùng với OpenFeign cho giao tiếp đồng bộ giữa các dịch vụ.
Ở tầng dữ liệu, hệ thống áp dụng chiến lược polyglot persistence với PostgreSQL cho dữ liệu
quan hệ, Redis cho cache và thao tác nguyên tử, Elasticsearch cho tìm kiếm toàn văn. Apache
Kafka đảm nhiệm vai trò message broker trung tâm cho toàn bộ giao tiếp bất đồng bộ. Keycloak
cung cấp giải pháp bảo mật tập trung, VNPAY xử lý thanh toán trực tuyến. Ở tầng giao diện,
Next.js được dùng để xây dựng storefront và Admin Panel. Toàn bộ hệ thống được đóng gói và
vận hành thông qua Docker Compose; môi trường production-like chạy trên AWS EC2, dùng Caddy
làm reverse proxy/HTTPS và GitHub Actions/GHCR cho quy trình build/push image. Prometheus,
Grafana và Zipkin tạo thành bộ công cụ quan sát toàn diện.

\section{Ngôn ngữ và nền tảng phát triển}
\label{section:3.2}

\subsection{Java 21}

Java 21 được chọn làm ngôn ngữ lập trình chính cho toàn bộ hệ thống. Đây là phiên bản Hỗ
trợ Dài hạn (LTS) mới nhất tại thời điểm phát triển, được Oracle cam kết hỗ trợ đến năm
2031. Java 21 mang lại nhiều cải tiến đáng kể so với các phiên bản LTS trước đó: Virtual
Threads (Project Loom) cho phép xử lý hàng triệu kết nối đồng thời với chi phí thấp hơn
nhiều so với mô hình thread truyền thống; Pattern Matching cải thiện cú pháp làm việc với
kiểu dữ liệu; Record Classes giảm đáng kể mã boilerplate cho các lớp DTO.

Đối chiếu với các ngôn ngữ JVM khác, Kotlin cung cấp cú pháp ngắn gọn hơn và null safety
tốt hơn nhưng cộng đồng và hệ sinh thái nhỏ hơn đáng kể so với Java, đặc biệt trong mảng
enterprise. Scala có hệ thống kiểu mạnh mẽ hơn nhưng đường cong học tập dốc và tốc độ biên
dịch chậm. Trong bối cảnh đồ án tốt nghiệp với yêu cầu ổn định, tài liệu phong phú và khả
năng tuyển dụng sau khi tốt nghiệp, Java 21 là lựa chọn tối ưu.

Hạn chế chính của Java trong bối cảnh microservices là thời gian khởi động tương đối chậm
và mức tiêu thụ bộ nhớ cao hơn so với Go hoặc Rust. Điều này được khắc phục một phần thông
qua cấu hình JVM memory cap trong Docker Compose production profile.

\subsection{Maven Multi-Module}

Toàn bộ mã nguồn được tổ chức theo cấu trúc Maven multi-module với một file \texttt{pom.xml}
cha quản lý tập trung các phiên bản dependency, plugin và cấu hình build chung. Cấu trúc
này mang lại ba lợi ích chính. Thứ nhất, đảm bảo tính nhất quán về phiên bản thư viện trên
toàn bộ 17 module --- tránh xung đột dependency khi các service sử dụng chung các thư viện
như Spring Boot, Spring Cloud, Jackson và Lombok. Thứ hai, module \texttt{common} chứa các
lớp dùng chung (\texttt{ApiResponse}, các lớp ngoại lệ chuẩn hóa, các DTO hợp đồng sự kiện)
mà tất cả service đều phụ thuộc vào, giúp tránh trùng lặp mã nguồn và đảm bảo tính nhất
quán trong giao tiếp. Thứ ba, Maven Wrapper (\texttt{mvnw}) được tích hợp vào dự án, cho
phép biên dịch toàn bộ hệ thống mà không cần cài đặt Maven thủ công.

So với Gradle --- một lựa chọn thay thế phổ biến với DSL linh hoạt hơn và tốc độ build
nhanh hơn nhờ incremental build và build cache --- Maven được chọn vì cấu trúc XML tường
minh, dễ đọc và phổ biến rộng rãi trong các dự án Spring Boot truyền thống. Với quy mô 17
module của đồ án, sự khác biệt về tốc độ build giữa Maven và Gradle không đáng kể.

\subsection{Spring Boot 3.5}

Spring Boot 3.5 là nền tảng phát triển chính cho toàn bộ 16 dịch vụ Java trong hệ thống
(13 business service và 3 infrastructure service). Spring Boot được chọn vì bốn lý do chính:
(1)~cơ chế auto-configuration loại bỏ phần lớn cấu hình thủ công, cho phép tập trung vào
logic nghiệp vụ; (2)~hệ sinh thái Spring phong phú cung cấp giải pháp cho hầu hết mọi nhu
cầu --- từ bảo mật (Spring Security), truy cập dữ liệu (Spring Data JPA, Spring Data Redis,
Spring Data Elasticsearch), đến messaging (Spring Kafka) và observability (Micrometer
Tracing); (3)~Actuator tích hợp sẵn các endpoint giám sát và health check, là
nền tảng cho toàn bộ hệ thống observability; (4)~khả năng embedded server (Netty cho
Gateway, Tomcat cho các service khác) giúp đóng gói mỗi service thành một JAR tự chứa, dễ
dàng container hóa.

Đối chiếu với các lựa chọn thay thế: Quarkus và Micronaut cung cấp thời gian khởi động
nhanh hơn và mức tiêu thụ bộ nhớ thấp hơn nhờ cơ chế compile-time dependency injection
thay vì runtime reflection, rất phù hợp cho môi trường serverless hoặc Kubernetes. Tuy
nhiên, Spring Boot vượt trội về độ hoàn thiện của hệ sinh thái, chất lượng tài liệu và
quy mô cộng đồng --- những yếu tố quan trọng đối với một đồ án tốt nghiệp có thời gian
giới hạn.

\subsection{Thư viện hỗ trợ}

Bên cạnh các framework chính, dự án sử dụng một số thư viện hỗ trợ quan trọng. Lombok
giảm đáng kể mã boilerplate cho các lớp entity, DTO và configuration thông qua các
annotation như \texttt{@Data}, \texttt{@Builder}, \texttt{@RequiredArgsConstructor}, cho
phép tập trung vào logic nghiệp vụ thay vì viết getter, setter và constructor thủ công.
Jackson đảm nhiệm việc tuần tự hóa và giải tuần tự hóa JSON trong toàn bộ hệ thống,
từ request/response của REST API đến payload của Kafka message. Flyway quản lý phiên
bản schema database thông qua các migration script SQL được đánh số thứ tự, đảm bảo
schema database luôn đồng bộ với phiên bản mã nguồn hiện tại trên mọi môi trường.

\section{Hệ sinh thái Spring Cloud}
\label{section:3.3}

\subsection{Spring Cloud Gateway}

Spring Cloud Gateway đóng vai trò API Gateway --- điểm vào duy nhất cho mọi yêu cầu từ
client đến hệ thống. Gateway được xây dựng trên nền tảng Spring WebFlux và Netty, cung cấp
mô hình xử lý bất đồng bộ, non-blocking, cho phép xử lý hàng nghìn kết nối đồng thời với
số lượng thread hạn chế.

Gateway được cấu hình thông qua các route định tuyến, mỗi route ánh xạ một đường dẫn
(\texttt{/api/products/**}) đến một dịch vụ đích (\texttt{lb://product-service}) thông qua
Eureka service discovery. Các filter được áp dụng trên từng route hoặc toàn cục để xử lý
các trách nhiệm cross-cutting: AuthHeaderFilter trích xuất thông tin người dùng từ JWT và
chèn vào header của request chuyển tiếp đến backend; GuestSessionFilter tạo và duy trì
session cho người dùng chưa đăng nhập khi thao tác với giỏ hàng; RateLimiterFilter áp dụng
Redis token bucket rate limiter cho các endpoint nhạy cảm. CORS được cấu hình tập trung tại
Gateway thông qua biến môi trường \texttt{CORS\_ALLOWED\_ORIGINS}, cho phép linh hoạt thay
đổi giữa môi trường phát triển và sản xuất.

So sánh với các giải pháp thay thế: Netflix Zuul (phiên bản 1.x dựa trên Servlet API
blocking, hiệu năng thấp hơn đáng kể so với reactive Gateway; phiên bản 2.x chưa bao giờ
được phát hành chính thức). Kong và Traefik là các API Gateway độc lập, mạnh mẽ và hiệu
năng cao, nhưng yêu cầu vận hành riêng biệt và không tích hợp sẵn với Spring Security/OAuth2
như Spring Cloud Gateway. Với hệ thống toàn Spring, Gateway là lựa chọn tự nhiên và tối ưu.

\subsection{Netflix Eureka --- Service Discovery}

Eureka Server đóng vai trò Service Registry --- nơi tất cả dịch vụ đăng ký địa chỉ của mình
khi khởi động. Mỗi service được cấu hình với \path|eureka.client.serviceUrl.defaultZone|
trỏ về Eureka Server tại \path|http://eureka:8761/eureka|. Khi API Gateway cần định tuyến
một request đến \path|lb://product-service|, nó truy vấn Eureka để lấy danh sách các
instance hiện có của product-service và thực hiện client-side load balancing.

Eureka hoạt động theo mô hình AP (ưu tiên Availability và Partition Tolerance theo CAP
Theorem), phù hợp với yêu cầu của hệ thống TMĐT --- nơi tính sẵn sàng của service registry
quan trọng hơn tính nhất quán tuyệt đối. Cơ chế self-preservation của Eureka ngăn việc xóa
hàng loạt registration khi có sự cố mạng tạm thời, đảm bảo hệ thống tiếp tục hoạt động.

Lựa chọn thay thế: Consul (HashiCorp) cung cấp health check mạnh mẽ hơn và hỗ trợ key-value
store có thể thay thế Config Server; Kubernetes Service + Ingress cung cấp service discovery
tự nhiên trong môi trường Kubernetes nhưng không khả dụng trong môi trường Docker Compose
mà đề tài hướng tới. Eureka được chọn vì tích hợp chặt chẽ với Spring Cloud, cấu hình tối
thiểu và đáp ứng tốt yêu cầu single-host của đồ án.

\subsection{Spring Cloud Config Server}

Config Server cung cấp cơ chế quản lý cấu hình tập trung cho toàn bộ hệ thống. Thay vì mỗi
service nhúng cấu hình trong file \texttt{application.yml} tĩnh hoặc phụ thuộc vào biến môi
trường rải rác, tất cả cấu hình được lưu trữ tập trung trên Config Server và được phục vụ
qua HTTP API. Mỗi service khi khởi động sẽ gọi đến Config Server để lấy cấu hình tương ứng
với tên service và profile đang chạy, cho phép thay đổi cấu hình tại một điểm duy nhất và
áp dụng cho tất cả instance của service mà không cần rebuild image.

Trong đồ án, Config Server sử dụng native profile (đọc từ classpath) thay vì Git backend,
phù hợp với quy mô đồ án và môi trường Docker. Cấu hình được tổ chức theo cấu trúc
\texttt{\{service-name\}-\{profile\}.yml}, cho phép mỗi service có cấu hình riêng cho môi
trường docker và local. Hạn chế của cách tiếp cận này là cần rebuild Config Server image
mỗi khi thay đổi cấu hình, không linh hoạt bằng Git backend hoặc các giải pháp quản lý cấu
hình động như Vault.

\subsection{OpenFeign --- Giao tiếp đồng bộ giữa các dịch vụ}

OpenFeign là thư viện HTTP client khai báo (declarative) cho phép giao tiếp giữa các dịch
vụ một cách đơn giản: lập trình viên chỉ cần định nghĩa một interface với các annotation mô
tả endpoint đích, OpenFeign tự động sinh implementation tại runtime. Cơ chế này giảm đáng
kể lượng code boilerplate so với việc sử dụng \texttt{RestTemplate} hoặc \texttt{WebClient}
thủ công, đồng thời tích hợp sẵn với Eureka để phân giải tên dịch vụ thành địa chỉ thực tế.

Trong hệ thống, OpenFeign được sử dụng cho các tương tác đồng bộ cần phản hồi ngay: cart-service
gọi product-service để xác minh sản phẩm; order-service gọi product-service, inventory-service
và voucher-service để kiểm tra trước khi tạo đơn; review-service gọi order-service để xác
minh quyền đánh giá. Các Feign client được cấu hình timeout và retry để xử lý tình huống
dịch vụ đích tạm thời không khả dụng.

\section{Hạ tầng dữ liệu}
\label{section:3.4}

\subsection{PostgreSQL 17}

PostgreSQL là hệ quản trị cơ sở dữ liệu quan hệ chính của hệ thống, được sử dụng cho 10
trong số 13 business service theo nguyên tắc database-per-service. Trên một instance
PostgreSQL duy nhất, mỗi service sở hữu một logical database riêng biệt (ví dụ:
\texttt{order\_db}, \texttt{payment\_db}, \texttt{product\_db}), đảm bảo cách ly dữ liệu
ở mức schema và cho phép mỗi service quản lý schema độc lập thông qua Flyway migration.

PostgreSQL được chọn thay vì MySQL vì bốn lý do. Thứ nhất, PostgreSQL hỗ trợ tốt hơn các
kiểu dữ liệu nâng cao như UUID (được sử dụng làm khóa chính cho hầu hết các bảng trong hệ
thống), JSONB cho dữ liệu bán cấu trúc và ARRAY cho các tập giá trị. Thứ hai, PostgreSQL
có hệ thống indexing mạnh mẽ hơn với GiST, GIN và BRIN, đặc biệt hữu ích cho các truy vấn
tìm kiếm và lọc phức tạp trong TMĐT. Thứ ba, MVCC (Multi-Version Concurrency Control) của
PostgreSQL xử lý tốt hơn các tình huống đọc-ghi đồng thời cao so với cơ chế khóa của MySQL
InnoDB. Thứ tư, PostgreSQL có tính tương thích cao với Flyway và Spring Data JPA.

\subsection{Redis 8}

Redis đảm nhiệm ba vai trò quan trọng trong hệ thống, tận dụng mô hình single-threaded và
khả năng thao tác trong bộ nhớ với độ trễ dưới mili-giây. Vai trò thứ nhất là \textbf{cache
phân tán}: product-service sử dụng Redis để cache thông tin sản phẩm, giảm tải cho PostgreSQL
và giảm độ trễ phản hồi cho các request đọc danh sách sản phẩm --- chiếm phần lớn lưu lượng
của hệ thống TMĐT. Vai trò thứ hai là \textbf{lưu trữ giỏ hàng}: cart-service lưu toàn bộ
giỏ hàng của người dùng trong Redis, với key được tạo từ userId hoặc sessionId, tận dụng
thời gian truy cập nhanh của Redis và cơ chế TTL để tự động dọn dẹp giỏ hàng bỏ hoang.
Vai trò thứ ba là \textbf{rate limiting}: API Gateway sử dụng Redis làm backend cho token
bucket rate limiter, lưu trữ và cập nhật số lượng token còn lại cho mỗi client.

So sánh với các lựa chọn thay thế: Memcached là giải pháp cache đơn giản hơn nhưng thiếu
các cấu trúc dữ liệu phong phú của Redis (Sorted Set cho ranking, Hash cho object, List cho
queue) và không hỗ trợ Lua scripting --- yếu tố then chốt cho atomic operation trong flash
sale. Hazelcast cung cấp cache phân tán embedded trong JVM, loại bỏ dependency network đến
Redis, nhưng tiêu tốn bộ nhớ heap của chính service và phức tạp hơn trong việc đồng bộ
trạng thái giữa các instance.

\subsection{Elasticsearch 8}

Elasticsearch được sử dụng làm search engine cho chức năng tìm kiếm toàn văn sản phẩm, triển
khai theo mô hình CQRS-lite. Trong khi product-service (với PostgreSQL) đóng vai trò write
model --- nơi quản trị viên tạo và cập nhật thông tin sản phẩm, search-service (với
Elasticsearch) đóng vai trò read model --- phục vụ các truy vấn tìm kiếm phức tạp từ phía
khách hàng. Hai mô hình được đồng bộ qua Kafka: mỗi khi sản phẩm được tạo, cập nhật hoặc
xóa, một sự kiện được phát hành và search-service tiêu thụ để cập nhật chỉ mục Elasticsearch
tương ứng.

Elasticsearch được chọn thay vì Apache Solr vì khả năng mở rộng theo chiều ngang tốt hơn,
REST API hiện đại và tích hợp tốt với hệ sinh thái Spring Data Elasticsearch. So với
PostgreSQL full-text search (\texttt{tsvector}), Elasticsearch vượt trội về tốc độ tìm kiếm
trên tập dữ liệu lớn, hỗ trợ tìm kiếm mờ (fuzzy search), gợi ý tự động (autocomplete) và
phân tích tần suất từ khóa --- những tính năng quan trọng đối với trải nghiệm tìm kiếm sản
phẩm trong TMĐT. Tuy nhiên, Elasticsearch tiêu tốn nhiều tài nguyên hơn (RAM và CPU) và
cần được cấu hình JVM heap cẩn thận để tránh ảnh hưởng đến các container khác trong môi
trường Docker Compose single-host.

\section{Apache Kafka --- Message Broker}
\label{section:3.5}

Apache Kafka được chọn làm message broker trung tâm, đảm nhiệm toàn bộ giao tiếp bất đồng
bộ giữa các dịch vụ trong hệ thống. Kafka vận hành ở chế độ KRaft (KRaft metadata mode) ---
không phụ thuộc vào Apache ZooKeeper, giảm đáng kể số lượng thành phần hạ tầng cần quản lý
và đơn giản hóa việc triển khai. Đây là cải tiến quan trọng của Kafka từ phiên bản 3.x,
đặc biệt phù hợp với môi trường triển khai single-host như đồ án này.

Kafka được chọn thay vì RabbitMQ vì ba ưu thế chính trong bối cảnh hệ thống TMĐT. Thứ nhất,
Kafka được thiết kế cho throughput cao với cơ chế lưu trữ tuần tự trên đĩa (sequential disk
I/O) và zero-copy transfer, cho phép xử lý hàng triệu thông điệp mỗi giây --- phù hợp với
lưu lượng lớn trong các sự kiện flash sale. Thứ hai, Kafka lưu trữ thông điệp bền vững trên
đĩa với thời gian lưu trữ có thể cấu hình (retention), cho phép consumer đọc lại thông điệp
từ quá khứ --- hữu ích cho các cơ chế reconciliation và khôi phục trạng thái. Thứ ba, cơ
chế partitioning và consumer group của Kafka cho phép xử lý song song và đảm bảo thứ tự
trong cùng một partition --- yếu tố quan trọng để tuần tự hóa yêu cầu flash sale.

Tuy nhiên, Kafka cũng có những hạn chế cần cân nhắc. So với RabbitMQ, Kafka có độ trễ
end-to-end cao hơn do cơ chế batch và polling, do đó kém phù hợp hơn cho các tương tác yêu
cầu phản hồi tức thời. Kafka cũng phức tạp hơn về mặt cấu hình và vận hành, tiêu tốn nhiều
tài nguyên hơn. Trong đồ án, các tương tác cần phản hồi ngay được xử lý qua REST/OpenFeign,
còn Kafka chỉ đảm nhiệm các luồng bất đồng bộ.

\section{Keycloak --- Bảo mật và quản lý định danh}
\label{section:3.6}

Keycloak 26 được chọn làm Identity Provider (IdP) cho toàn bộ hệ thống, chịu trách nhiệm
quản lý người dùng, xác thực và cấp phát token. Keycloak là giải pháp mã nguồn mở, được
phát triển bởi Red Hat, hỗ trợ đầy đủ các giao thức chuẩn công nghiệp: OAuth 2.0, OpenID
Connect và SAML 2.0. Với Keycloak, hệ thống không cần tự triển khai logic quản lý người
dùng, xác thực mật khẩu, quản lý phiên làm việc hay làm mới token --- tất cả được Keycloak
xử lý một cách chuẩn hóa và an toàn.

Keycloak được lựa chọn thay vì các giải pháp thay thế vì những lý do sau. So với việc tự
xây dựng hệ thống xác thực (Spring Security + database người dùng tự quản), Keycloak cung
cấp sẵn các tính năng bảo mật phức tạp đã được kiểm chứng: hỗ trợ đa yếu tố xác thực
(MFA), chính sách mật khẩu, brute-force detection, social login (Google, Facebook) và quản
trị tập trung qua giao diện web. So với Auth0 hoặc Okta --- các giải pháp Identity-as-a-Service
thương mại, Keycloak là mã nguồn mở và tự lưu trữ (self-hosted), không phụ thuộc vào bên
thứ ba, không giới hạn số lượng người dùng và phù hợp với môi trường học thuật không có
ngân sách. So với Apache Shiro hoặc CAS, Keycloak có hệ sinh thái tích hợp Spring Boot/Security
phong phú hơn và cộng đồng hỗ trợ lớn hơn.

Trong kiến trúc bảo mật của hệ thống, Keycloak đảm nhiệm ba chức năng chính. Chức năng thứ
nhất là \textbf{cấp phát token}: sau khi người dùng đăng nhập thành công (qua giao diện đăng
nhập của Keycloak hoặc qua API của identity-service), Keycloak cấp một bộ token gồm Access
Token (JWT, thời hạn 5 phút), Refresh Token (thời hạn 30 phút) và ID Token. Access Token
được ký bằng thuật toán RS256 với private key của Keycloak; public key được công khai tại
endpoint JWKS (\texttt{/realms/ecommerce/protocol/openid-connect/certs}) để API Gateway xác
minh. Chức năng thứ hai là \textbf{làm mới token}: khi Access Token hết hạn, client sử dụng
Refresh Token để nhận Access Token mới mà không cần đăng nhập lại. Chức năng thứ ba là
\textbf{quản trị tập trung}: quản trị viên có thể tạo, sửa, khóa người dùng, quản lý vai
trò (ROLE\_USER, ROLE\_ADMIN) và theo dõi phiên làm việc qua Keycloak Admin Console tại
\texttt{http://localhost:8180}.

Hạn chế của Keycloak trong bối cảnh đồ án: Keycloak là một ứng dụng Java nặng, tiêu tốn
đáng kể tài nguyên (RAM và CPU) trên môi trường single-host. Điều này được quản lý thông
qua giới hạn bộ nhớ trong Docker Compose và chấp nhận như một đánh đổi để có được giải pháp
bảo mật toàn diện.

\section{VNPAY --- Cổng thanh toán trực tuyến}
\label{section:3.7}

VNPAY được chọn làm cổng thanh toán cho chức năng thanh toán trực tuyến của hệ thống. Đây
là một trong những cổng thanh toán hàng đầu tại Việt Nam, hỗ trợ đa dạng phương thức thanh
toán bao gồm thẻ ATM nội địa, thẻ tín dụng quốc tế (Visa, Mastercard, JCB), ví điện tử và
QR code. Đối với đồ án tốt nghiệp, VNPAY cung cấp môi trường sandbox miễn phí với đầy đủ
API, tài liệu và thẻ test, cho phép kiểm thử toàn bộ luồng thanh toán mà không cần giao
dịch thật.

Quy trình thanh toán qua VNPAY được thiết kế theo ba bước. Bước thứ nhất, payment-service
tạo một URL thanh toán bằng cách gửi yêu cầu đến VNPAY với các tham số: mã đơn hàng, số
tiền, mô tả đơn hàng, địa chỉ IP của khách hàng, ngân hàng (tùy chọn) và ngôn ngữ hiển thị.
Tất cả tham số được sắp xếp theo thứ tự alphabet và ký với HMAC-SHA512 sử dụng secret key
được VNPAY cấp, tạo ra một chữ ký số (\texttt{vnp\_SecureHash}) đảm bảo tính toàn vẹn và
chống giả mạo. Bước thứ hai, khách hàng được chuyển hướng đến trang thanh toán của VNPAY
để nhập thông tin thẻ và xác nhận thanh toán. Bước thứ ba, sau khi thanh toán hoàn tất
(thành công hoặc thất bại), VNPAY gọi đến Return URL đã đăng ký kèm theo kết quả thanh toán
đã được ký, payment-service xác minh chữ ký và cập nhật trạng thái thanh toán. Song song,
VNPAY cũng gửi IPN (Instant Payment Notification) đến một endpoint riêng để đảm bảo kết quả
được ghi nhận ngay cả khi khách hàng đóng trình duyệt trước khi được chuyển hướng về Return
URL.

Lựa chọn thay thế cho VNPAY bao gồm Momo, ZaloPay và PayPal. Momo và ZaloPay có quy trình
tích hợp tương tự nhưng tài liệu sandbox và thẻ test ít phong phú hơn cho mục đích học
thuật. PayPal phổ biến toàn cầu nhưng không hỗ trợ tốt các ngân hàng nội địa Việt Nam.
VNPAY được chọn vì tính phổ biến tại Việt Nam, tài liệu tiếng Việt đầy đủ và sandbox dễ
tiếp cận.

Hạn chế: phạm vi đồ án chỉ dừng ở môi trường sandbox, chưa kiểm thử với giao dịch thật.
Trong môi trường production, cần xử lý thêm các tình huống như hoàn tiền, truy vấn giao
dịch và đối soát định kỳ.

\section{Hệ thống quan sát --- Observability Stack}
\label{section:3.8}

\subsection{Prometheus và Grafana --- Giám sát số liệu}

Prometheus được chọn làm hệ thống thu thập và lưu trữ metrics cho toàn bộ hệ thống. Đây là
giải pháp giám sát mã nguồn mở, được thiết kế đặc biệt cho môi trường động và microservices
với cơ chế pull-based: thay vì chờ service đẩy metrics về, Prometheus chủ động kéo (scrape)
dữ liệu từ các endpoint \texttt{/actuator/prometheus} của từng service theo chu kỳ cấu hình
(15 giây trong đồ án). Cơ chế này giúp Prometheus tự phát hiện service mới hoặc service
ngừng hoạt động mà không cần cấu hình thủ công, đồng thời giảm tải cho service so với mô
hình push-based.

Prometheus sử dụng ngôn ngữ truy vấn PromQL mạnh mẽ, cho phép tính toán các chỉ số phức tạp
như latency percentile (p50, p95, p99), tỷ lệ lỗi theo thời gian và tốc độ tăng trưởng của
một chỉ số. Dữ liệu được lưu trữ theo mô hình time-series với khả năng nén hiệu quả, phù
hợp cho việc lưu trữ dài hạn trên môi trường single-host với tài nguyên hạn chế.

Grafana đảm nhiệm vai trò trực quan hóa dữ liệu từ Prometheus. Các dashboard được provision
tự động thông qua cơ chế file-based provisioning của Grafana, bao gồm \path|jvm-overview|,
\path|spring-boot-overview| và \path|ecommerce-saga-overview|. Ba dashboard này tập trung
vào các chỉ số JVM, request/latency/error rate của Spring Boot service và trạng thái các
luồng Saga cốt lõi. Grafana được chọn thay vì Kibana vì tích hợp tốt hơn với Prometheus
(Kibana chủ yếu dành cho Elasticsearch) và cộng đồng dashboard template phong phú hơn.

\subsection{Zipkin --- Truy vết phân tán}

Zipkin được chọn làm hệ thống distributed tracing, cho phép theo dõi hành trình của một
request qua nhiều service. Zipkin là giải pháp mã nguồn mở, nhẹ và dễ triển khai --- trong
đồ án, Zipkin chạy như một container độc lập với storage in-memory, không yêu cầu cơ sở dữ
liệu backend phức tạp. Zipkin tuân thủ chuẩn W3C Trace Context, đảm bảo khả năng tương tác
với các thư viện tracing khác nhau trong hệ sinh thái.

Mỗi service trong hệ thống được tích hợp Micrometer Tracing (trước đây là Spring Cloud
Sleuth) để tự động sinh Trace ID và Span ID cho mỗi request, đồng thời gửi dữ liệu trace
về Zipkin server qua HTTP. Khi cần điều tra một request cụ thể --- ví dụ một đơn hàng thất
bại --- kỹ sư có thể tìm kiếm theo Trace ID trên Zipkin UI và xem toàn bộ chuỗi span, thời
gian xử lý tại từng service và các tag nghiệp vụ kèm theo. Điều này đặc biệt hữu ích cho
việc gỡ lỗi các luồng Saga phức tạp.

Lựa chọn thay thế: Jaeger (Uber) cung cấp hiệu năng cao hơn và hỗ trợ storage backend linh
hoạt hơn (Cassandra, Elasticsearch), phù hợp với hệ thống production quy mô lớn. Tuy nhiên,
Zipkin nhẹ hơn, dễ triển khai hơn và tích hợp sẵn trong Spring Boot ecosystem thông qua
Micrometer --- phù hợp hơn với quy mô đồ án.

\subsection{Mailpit --- Kiểm thử email}

Mailpit được chọn làm SMTP server cho môi trường phát triển, thay thế cho các dịch vụ email
thật như Gmail SMTP hoặc SendGrid. Mailpit bắt giữ tất cả email được gửi từ notification-service
và hiển thị chúng qua giao diện web tại \texttt{http://localhost:8025}, cho phép kiểm tra nội
dung email xác nhận đơn hàng, email hủy đơn và email thông báo flash sale mà không cần gửi
email thật ra bên ngoài. Trong môi trường production, Mailpit được thay thế bằng SMTP server
thật thông qua cấu hình biến môi trường.

\section{Next.js, AWS EC2 và CI/CD}
\label{section:3.frontend-deployment}

\subsection{Next.js --- Storefront và Admin Panel}

Next.js được chọn cho tầng giao diện vì hỗ trợ App Router, Server Components, Server Actions
và khả năng kết hợp linh hoạt giữa render phía server và tương tác phía client. Trong hệ thống,
Next.js phục vụ hai nhóm màn hình: storefront cho khách hàng và Admin Panel cho quản trị viên.
Các request từ trình duyệt đi qua lớp BFF/proxy của frontend; access token được lưu trong
httpOnly cookie và chỉ được gắn vào header \texttt{Authorization} khi server-side code gọi API
Gateway. Cách tổ chức này giảm nguy cơ lộ token ở phía client và phù hợp với mô hình Gateway
đóng vai Resource Server.

Admin Panel sử dụng route segment \texttt{/admin/*}, được bảo vệ bởi hai lớp: middleware kiểm
tra cookie/JWT và role \texttt{ROLE\_ADMIN}, sau đó layout phía server gọi \texttt{getAdminSession}
để chặn truy cập khi token thiếu hoặc không đủ quyền. Các mutation như tạo sản phẩm, điều chỉnh
tồn kho, cập nhật đơn hàng, quản lý voucher, flash-sale và content được thực hiện bằng Server
Actions, nhờ đó token không cần xuất hiện trong JavaScript chạy trên trình duyệt.

\subsection{AWS EC2, Caddy và GitHub Actions}

Môi trường triển khai của đồ án là một máy ảo AWS EC2 single-host, phù hợp với giới hạn tài
nguyên của đồ án và vẫn đủ để kiểm chứng các đặc tính vận hành chính. Caddy được dùng làm
reverse proxy vì có khả năng tự động cấp/gia hạn chứng chỉ TLS qua Let's Encrypt và cấu hình
đơn giản cho các subdomain dạng \path|nip.io|. Trên môi trường production-like, Grafana được
đổi sang cổng host \path|3001| để tránh xung đột với frontend ở cổng \path|3000|.

GitHub Actions đảm nhiệm quy trình build và publish Docker image lên GitHub Container Registry
(GHCR). File \path|docker-compose.prod.yml| sử dụng các image \path|ghcr.io/...| thay cho
local build, đồng thời đặt \path|mem_limit| và \path|mem_reservation| cho từng container
để phù hợp tài nguyên của EC2. Quy trình này chưa phải GitOps/Kubernetes pipeline hoàn chỉnh,
nhưng đáp ứng mục tiêu production-like deployment trong phạm vi đồ án.

\section{Docker Compose --- Đóng gói và triển khai}
\label{section:3.9}

Backend, frontend và các thành phần hạ tầng của hệ thống được định nghĩa và điều phối thông
qua Docker Compose. Mỗi service được mô tả trong file \path|docker-compose.yml| với đầy đủ thông tin: image
hoặc Dockerfile, cổng, biến môi trường, volume, network, healthcheck, resource limits và
mối quan hệ khởi động (\texttt{depends\_on}).

Docker Compose được chọn thay vì Kubernetes vì ba lý do. Thứ nhất, Docker Compose sử dụng
cú pháp YAML đơn giản, dễ đọc và dễ cấu hình. Thứ hai, Docker Compose phù hợp với mô hình triển
khai single-host mà đề tài hướng tới, trong khi Kubernetes được thiết kế cho môi trường
multi-node cluster và đòi hỏi kiến thức vận hành phức tạp hơn nhiều. Thứ ba, Docker Compose
tích hợp sẵn trong Docker Desktop, không yêu cầu cài đặt thêm công cụ, giúp giảm rào cản
tiếp cận cho người học.

Hệ thống sử dụng Docker Compose profile để tách biệt cấu hình giữa môi trường phát triển
và môi trường production. File \texttt{docker-compose.yml} định nghĩa cấu hình cơ sở dùng
chung; file \texttt{docker-compose.prod.yml} kế thừa và ghi đè các thiết lập cho production
như GHCR image, \texttt{mem\_limit}, \texttt{mem\_reservation}, cổng Grafana \texttt{3001}
và CORS allowed origins. Cách tiếp cận này tránh trùng lặp cấu hình và đảm bảo môi trường
development không bị ảnh hưởng bởi các ràng buộc của production.

\section{Công cụ kiểm thử}
\label{section:3.10}

\subsection{JUnit 5 và Testcontainers}

JUnit 5 là nền tảng kiểm thử chính cho toàn bộ hệ thống, được tích hợp sẵn trong Spring
Boot starter test. JUnit 5 mang lại nhiều cải tiến so với JUnit 4: mô hình extension
model linh hoạt hơn, hỗ trợ lambda trong assertion, tổ chức test theo class lồng nhau
(\texttt{@Nested}) và parameterized test mạnh mẽ hơn. Kết hợp với Mockito, JUnit 5 cho
phép mock các dependency bên ngoài (repository, Feign client, Kafka template) để kiểm
thử logic nghiệp vụ một cách cô lập.

Testcontainers là thư viện hỗ trợ kiểm thử tích hợp với cơ sở hạ tầng thật: thay vì sử
dụng H2 in-memory database --- vốn không tương thích hoàn toàn với PostgreSQL, Testcontainers
khởi tạo một container PostgreSQL thật trong quá trình chạy test, đảm bảo môi trường kiểm
thử giống hệt môi trường production. Testcontainers cũng hỗ trợ khởi tạo container Kafka,
Redis và Elasticsearch, mặc dù trong đồ án chủ yếu được sử dụng cho PostgreSQL.

\subsection{k6 --- Kiểm thử hiệu năng}

k6 được chọn làm công cụ kiểm thử hiệu năng cho hệ thống. Đây là công cụ mã nguồn mở, sử
dụng JavaScript làm ngôn ngữ kịch bản, cho phép viết các kịch bản kiểm thử phức tạp với
logic tùy chỉnh (tạo user, lấy token, đặt hàng, kiểm tra kết quả). k6 hỗ trợ nhiều mô
hình tải: constant load (duy trì số lượng VU cố định), ramping load (tăng dần số lượng
VU) và spike test (tăng đột biến). Kết quả kiểm thử được xuất ra dưới dạng JSON với các
chỉ số chi tiết: request rate, latency percentile, error rate, checks pass/fail --- các
chỉ số này được sử dụng làm bằng chứng định lượng trong Chương~\ref{chapter:SolutionAndContribution}.

So với JMeter --- công cụ kiểm thử hiệu năng truyền thống với giao diện GUI, k6 ưu việt
hơn ở khả năng viết kịch bản dưới dạng code (Infrastructure as Code), tích hợp CI/CD
dễ dàng hơn và tiêu tốn ít tài nguyên hơn khi chạy. Với yêu cầu mô phỏng 500 VU đồng
thời cho flash sale spike test, k6 là lựa chọn phù hợp.
